# Prompt Evaluation Framework

A small, practical toolkit for comparing prompt versions against a shared set
of test cases using explicit, explainable scoring criteria — **accuracy,
relevance, clarity, consistency, and instruction-following**.

Built by Amir Sadat as a hands-on prompt engineering project: not a wrapper
around an LLM API, but a working harness for the part of prompt engineering
that's easy to skip — deciding, in a repeatable and defensible way, whether
prompt B is actually better than prompt A.

## Why this exists

"I improved the prompt" is only a claim you can trust if you measured
something before and after. This framework runs several versions of the same
prompt against the same inputs, scores every output on five criteria, and
produces a ranked comparison — so prompt iteration has a feedback loop
instead of vibes.

## How it works

1. **Prompt versions** are plain text files, each paired with a small set of
   declared structural constraints (max sentences, max words, whether jargon
   is allowed, a required opening phrase). See
   [`examples/prompts/prompts.json`](examples/prompts/prompts.json).
2. **Test cases** are input passages plus a short list of "key facts" a good
   response should include. See
   [`examples/test_cases.json`](examples/test_cases.json).
3. A **backend** turns `(prompt, input)` into generated text. The bundled
   [`MockBackend`](pef/backends.py) is a deterministic, offline stand-in — no
   API key needed — that performs real extractive summarization and honors
   each prompt's declared constraints, so the whole pipeline is testable and
   reproducible without any external dependency. A thin `OpenAIBackend` is
   included for pointing the same harness at a real model later.
4. Each prompt version is run against every test case, **multiple times per
   case**, and the outputs are scored against all five criteria.
5. A **report** (Markdown table + JSON) ranks prompt versions by a composite
   score.

Every scorer is rule-based and inspectable — there's no hidden "LLM judge"
black box making the call. See [Evaluation Criteria](#evaluation-criteria)
for exactly how each score is computed.

## Evaluation Criteria

| Criterion | What it measures | How it's computed |
|---|---|---|
| **Accuracy** | Did the output retain the required facts? | Fraction of a test case's declared `key_facts` phrases found in the output. |
| **Relevance** | Does the output actually engage with the source material? | Fraction of the source text's top keywords (frequency-based, stopwords excluded) that appear in the output. |
| **Clarity** | Is the output easy for a general audience to read? | A from-scratch Flesch Reading Ease approximation (sentence length + syllables per word), normalized to 0–1. |
| **Consistency** | Does the prompt produce stable output across repeated runs? | Average pairwise word-overlap (Jaccard similarity) across N repeated generations for the same input. |
| **Instruction-following** | Does the output obey the prompt's own declared constraints? | Fraction of that prompt version's declared constraints (sentence/word limits, jargon-free, required prefix) that are satisfied. |

The composite score is an equal-weighted average of all five by default;
`PromptEvaluator(weights={...})` accepts custom weights if some criteria
matter more for a given task.

**A known limitation, stated up front:** accuracy and relevance here are
literal phrase/keyword matches, not semantic similarity. A jargon-free
rewrite that paraphrases a fact can score lower on accuracy than a jargon-
heavy one that quotes it verbatim, even if a human would judge both as
equally correct. This is the same trade-off classic ROUGE-style metrics
have — cheap and reproducible, but blind to paraphrase. Swapping in a
semantic-similarity or LLM-judge scorer for accuracy/relevance is a natural
next step; the criteria in [`pef/criteria.py`](pef/criteria.py) are
independent functions specifically so any one of them can be replaced
without touching the rest of the framework.

## Example prompts

Three versions of the same task — *"summarize a technical passage for a
non-technical audience"* — with progressively more explicit instructions:

- **`v1_naive`** — `"Summarize the following text."` No stated length,
  audience, or jargon guidance.
- **`v2_role_primed`** — role-primed, with an explicit sentence/word limit
  and a plain-language requirement.
- **`v3_few_shot_strict`** — adds a one-shot example and a stricter word
  budget, and requires the response to start with `"In short,"`.

Full text: [`examples/prompts/`](examples/prompts/).

## Sample evaluation results

Generated by the bundled `MockBackend` (3 runs per test case, fully
reproducible — re-run `python -m pef.cli evaluate` and you'll get the same
numbers). Full detail, including every generated output, is in
[`results/sample_results.md`](results/sample_results.md).

| Rank | Prompt | Composite | Accuracy | Relevance | Clarity | Consistency | Instruction Following |
|---|---|---|---|---|---|---|---|
| 1 | v1_naive | 73% | 50% | 79% | 45% | 90% | 100% |
| 2 | v2_role_primed | 67% | 25% | 71% | 49% | 90% | 100% |
| 3 | v3_few_shot_strict | 64% | 17% | 54% | 52% | 100% | 100% |

**What this actually shows:** the naive prompt wins on composite score, and
that's a real, useful finding rather than a bug in the demo. `v1` is allowed
to produce more sentences, so it simply retains more of the source's key
facts and keywords (higher accuracy/relevance). `v2` and `v3` trade some of
that factual coverage for a genuinely more readable, more consistent
output — clarity and consistency both improve monotonically from `v1` to
`v3`. Neither prompt "wins" outright; they sit at different points on a
length-vs-fidelity trade-off curve.

The actionable next step this evaluation points to: `v2`/`v3`'s word budgets
are tighter than this task's factual density supports. The next iteration
would loosen `max_words` slightly while keeping the plain-language and
audience framing, then re-run the same evaluation to confirm accuracy
recovers without giving back the clarity gains. That loop — change the
prompt, re-run the same test cases, compare — is the actual point of this
project.

## Usage

### Install

```bash
pip install -r requirements.txt   # pytest only; the framework itself has zero required deps
```

### Run the test suite

```bash
python -m pytest -v
```

### Run an evaluation

```bash
python -m pef.cli evaluate
```

Writes `results/sample_results.md` and `results/sample_results.json`, and
prints the Markdown table to stdout. Useful flags:

```bash
python -m pef.cli evaluate \
  --prompts examples/prompts/prompts.json \
  --test-cases examples/test_cases.json \
  --output-dir results \
  --runs-per-case 3 \
  --backend mock   # or: openai (requires `pip install openai` + OPENAI_API_KEY)
```

### Bring your own prompts and test cases

Drop new `.txt` prompt files next to the existing ones, add an entry to
`prompts.json` with their constraints, and add test cases (input text + key
facts) to `test_cases.json`. No code changes required.

### Optional: interactive dashboard

```bash
pip install -r requirements-optional.txt
streamlit run app.py
```

## Project structure

```
pef/
  criteria.py    # the 5 scoring functions
  backends.py     # MockBackend (offline, deterministic) + OpenAIBackend (optional)
  evaluator.py    # orchestrates running + scoring
  loaders.py      # reads the prompts.json / test_cases.json manifests
  report.py       # Markdown / JSON / pandas report builders
  cli.py          # `python -m pef.cli evaluate`
examples/
  prompts/        # the 3 example prompt versions + manifest
  test_cases.json # 3 example test cases with key facts
results/
  sample_results.md / .json   # committed output of the example run above
tests/            # pytest suite (27 tests) covering every scorer,
                  # the mock backend's constraint handling, and end-to-end runs
app.py            # optional Streamlit results viewer
```
